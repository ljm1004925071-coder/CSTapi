FieldSource Object

Use this object to define a field source for the simulation.

  • Methods: General

  • Methods: Field Import

  • Example

  • Convenience Methods

Methods: General

Reset ()

Resets all internal settings to their default values.

 

Name ( string name )

Specifies the name for the current distribution.

 

FileName  ( string name )

Specifies the name of the field source data file. Available for import are CST Microwave Studio FSM files (*.fsm), CST Microwave Studio hexanhedral field monitors (*.m3d), CST Mphysics Studio displacement files (*.fic), CST Cable Studio / CST PCB Studio RSD files (*.rsd), or Sigrity® NFD files (*.nfd).

 

Delete ( string name )

Deletes an existing field source, which is specified by the name.

 

Id ( integer uniqueID)

Set the unique ID of the field source which was generated using GetNextId.

 

GetNextId () integer

Returns the next free unique ID for a new field source. The filename has to be set and match the filename of the new field source. Set filename with FileName.

DeleteAll

Delete all field sources.

 

Rename  ( string oldname, string newname )

Renames an existing field source from oldname to newname.

 

Read

Activates the import, which has to be previously specified using the FileName method. This method is used for the following file types:  *.rsd, *.fsm, and *.nfd.

Methods: Field Import

ProjectPath (string path)

Specifies the project the field import is taken from. This may be an absolute path (UseRelativePath is set to "False") or a path relative to the current project file (UseRelativePath is set to "True").

 

UseRelativePath (bool flag)

Defines if the ProjectPath is given with a relative or absolute path. In the former case ProjectPath must specify a location relative to the current project file.

 

ResultSubDirectory  (string dirname)

Specifies the name of the subdirectory within the Result directory where the solver results can be found. For example, IR-Drop solver writes the archive with thermal losses into Result\IRDROPRESULTS. In this case, the name of subdirectory must be specified by .ResultSubDirectory "IRDROPRESULTS".

 

SourceName (enum FieldType)

Defines which field should be imported from the project specified by ProjectPath. The value of FieldType must be "Temperature", "Displacement", "Force Density", "Nodal Forces" or "Thermal Losses".

 

FieldMonitorName (enum name)

If the project specified by ProjectPath contains a transient thermal solution, use this command to specify the monitor from which the distribution should be imported. name may have the following values:

”Initial Solution”

the initial temperature distribution is imported.

”Stationary Solution”

if the source project contains a solution made by the stationary thermal solver, this solution will be imported.

Name of a 3D temperature monitor

one of the temperature distributions saved by this monitor is imported. The time frame is then specified in TimeValue or by UseLastTimeFrame.

 

ResultSubDirectory  (string dirname)

Specifies the name of the subdirectory within the Result directory where the solver results can be found. For example, IR-Drop solver writes the archive with thermal losses into Result\IRDROPRESULTS. In this case, the name of subdirectory must be specified by .ResultSubDirectory "IRDROPRESULTS".

 

ImportSettings (enum key, enum value)

Specifies additional field import settings in the form of a (key, value) correspondence. The settings are generally used to define and import an external field which could be possibly converted and / or processed before it can be used as valid source.

The available keys belong to the property enumeration enum { "FormatSource" , "FormatTarget" , "NastranFilename" , "TemperatureFilename" , "GeometryUnit" , "TemperatureUnit" , "WorkingDir" }.

Among all keys only "FormatSource" is always mandatory. The remaining keys are generally optional and their use strictly depends on the value selected for the "FormatSource" key and eventually on:

  •  the value set with the key "FormatTarget".

  • the field type selected with the method .SourceName.

  • the monitor type selected with the method .FieldMonitorName.

 

The available value(s) for the key "FormatSource" belong to the set enum { "Abaqus" , "Tet" , "Hex" , "CHT"}.

  • The value "Abaqus" specifies a distribution generated by Abaqus. The distribution is defined by means of a Nastran mesh with volumetric CHEXA elements (i.e. solid elements with eight corner GRIDs and six faces) together with an Abaqus Report File defining the nodal field at the grid positions.

  • The value "Tet" specifies the typical internal CST Studio Suite exchange format using tetrahedral mesh based solvers.

  • The value "Hex" specifies the typical internal CST Studio Suite exchange format using hexahedral mesh based solvers.

  • The value "CHT" specifies the typical internal CST Studio Suite exchange format using the conjugate heat transfer (CHT) solver.

 

At the current stage the only external field supported is a temperature distribution generated by Abaqus. This field can be imported into CST Studio suite using a tetrahedral based mesh conversion. This leads to the following settings:

.SourceName ( "Temperature" )

.FieldMonitorName ( "" )

.ImportSettings ( "FormatSource" , "Abaqus" )

.ImportSettings ( "FormatTarget", "Tet" )

 

This combination requires additional information to be provided by means of the .ImportSettings command as explained below:

  • "WorkingDir" - The value for this key is a ( string Directory ) specifying the folder where the Abaqus source files are expected. Directory is either an absolute path or a path relative to the <Project>/Temp subdirectory (here <Project> is the CST Studio Suite project).

  • "NastranFilename" - The value for this key is a ( string Filename ) specifying the file name of the mesh file generated by Abaqus in Nastran format. The file has typically the .inp or .nas extension and should contain a volumetric mesh with CHEXA elements. The mesh can be directly exported by Abaqus.

  • "TemperatureFilename" - The value for this key is a ( string Filename ) specifying the file name of the temperature scalar distribution generated by Abaqus. The file has typically the .rpt extension and is an ASCII file containing the nodal temperatures. Also this output can be directly generated in Abaqus using its Report functionality. The temperatures have to be written selecting "Unique Nodal" as Position and the field "NT11" has to be exported as output variable. It follows that the file contains at least the columns named "Node Label" and "NT11".

  • "GeometryUnit" - The value for this key is a ( enum GeomUnit ) specifying the geometrical unit adopted in the Nastran mesh file. The available units are the same specified with the .Geometry method in the Units object (see the Methods for Geometric Units page).

  • "TemperatureUnit" : the value for this key is a ( enum TempUnit ) specifying the temperature unit used in the report temperature file. The available units are the same specified with the .TemperatureUnit method in the Units object (see the Units Object page).

 

TimeValue (double time)

If a single temperature field is imported from a transient temperature monitor specified in FieldMonitorName, use this command to specify from which time point the temperature distribution must be imported. Parameter time must be specified in seconds. This command is ignored if UseLastTimeFrame is set to "True".

 

UseLastTimeFrame (bool flag)

If a single temperature field is imported from a transient temperature monitor specified in FieldMonitorName, use this command to select the last frame saved by the monitor.

 

UseCopyOnly (bool flag)

Defines if the internal copy of the field import should be used only. If this setting is activated, the imported field is not updated anymore, even if it has been changed in the source project.

 

CreateFieldImport

Creates a temperature, displacement, thermal loss or static force field import from a project, which needs to be specified with the ProjectPath method.

 

Example

' Obtain a free unique ID for "test.rsd"

Dim sID as String

 

With FieldSource

    .Reset

    .FileName ("test.rsd")

    fsID = .GetNextId

    .Reset

End With

 

' Define a field source with an evaluated ID

With FieldSource

    .Reset

    .Name ("CD1")

    .FileName ("test.rsd")

    .Id "42"

    .Read

End With

 

' Import a single (final) temperature distribution from a transient thermal project

With FieldSource

    .Reset

    .Name ("field1")

    .Id ("59")

    .ProjectPath ("Thermal TD.cst")

    .UseRelativePath ("True")

    .SourceName ("Temperature")

    .FieldMonitorName ("temp (t=0.0..end(100))")

    .UseLastTimeFrame ("True")

    .UseCopyOnly ("False")

    .CreateFieldImport

End With

 

'Delete the FieldSource

FieldSource .Delete ("CD1")

 

Activates the import, which has to be previously specified using the FileName method. This method is used for the following file types:  *.rsd, *.fsm, and *.nfd.

Convenience Methods

CreateFieldImportFromAbaqus( string name , string directory , string NastranFilename , string DistributionFilename ,  enum GeomUnit ,  enum TempUnit )

This is a convenience method to generate a complete and working field import definition using an external temperature distribution calculated by Abaqus. The method can be executed as control macro with the VBA macro editor and it stores a new command entry in the project history list filled with all information and the correct .ImportSettings commands (see paragraph Methods: Field Import). The item also appears in the project tree under Field Sources (please note that the generated item is at the moment not editable using the CST Studio Suite dialog but only working on the VBA History).

The name parameter is the name given to the created field source import.

The directory parameter is the folder where the Abaqus data are expected. It can be either an absolute path or a path relative to the <Project>/Temp subdirectory (here <Project> is the CST Studio Suite project).

NastranFilename and DistributionFilename correspond to the mesh output in Nastran format (extension .nas or .inp) and to the temperature field report (extension .rpt) generated by Abaqus.

Finallly GeomUnit and TempUnit specify the units for the geometry and the temperature to be used interpreting the Nastran mesh and the Temperature field report, respectively.

Please note that temperature field imports generated by this method are currently limited to use with the transient FIT-TD-solver.

 

CreateReverberationChamberFieldImport( string f_min, string f_max, string working_volume_x_min, string working_volume_y_min, string working_volume_z_min, string working_volume_x_max, string working_volume_y_max, string working_volume_z_max, string spacer, bool use_linear_samples_or_step,  string lin_freq_samples, string lin_freq_step, bool use_log_samples, string ext_freq_file, string monte_carlo_iters, string seed, string num_waves, bool generate_field_source, bool delete_previous_results, bool user_probes)

This is a convenience method to generate a complete and working field import simulating a reverberation chamber. The method generates a set of Monte Carlo iterations with randomly oriented and phase shifted plane waves sampled on the unit sphere. Together with the field sources some statistics (average and standard deviation) of the field peak value are produced, in order to asses the uniformity of the resulting field.

For further information about the usage of the macro and its results, please refer to the Solver / Reverberation Chamber pre loaded macro page where an in depth description is available.

The f_min and f_max parameters specify the minimum and maximum frequency range (in user unit) of the generated field sources.

The working_volume_x_min, working_volume_y_min, working_volume_z_min and their _max counterpart specify the working volume (in user unit). To be valid this bounding box should enclose a positive volume.

A spacer is introduced between the working volume and the generated field source. This is expressed in term of a layer number (spacer), where each layer´s thickness is evaluated as $\lambda_{min} / 15$, $\lambda_{min}$ being the minimal simulation wavelength according to the frequency settings (see below).

 

The frequency  samples distribution is controlled by the following parameters:

use_linear_samples_or_step specifies a set of uniformly distributed frequencies. If the flag is true, the samples are defined as a total number by the parameter lin_freq_samples. If instead the flag is false, the samples are spaced apart according to the parameter lin_freq_step.

With use_log_samples logarithmically distributed samples are added. The logarithmic sampling is not uniform but follows instead the IEC 61000-4-21 standard (for more information please refer to the Solver / Reverberation Chamber page).
Finally with ext_freq_file further samples can be added as specified in an external file. This file just contains a single column with the requested frequencies. The frequencies are generally assumed in user unit. This behavior can be overwritten by means of a header command, with the syntax #freq_scale:my_freq_scale, where my_freq_scale specifies the file frequency unit.

The monte_carlo_iters parameter controls the number of Monte Carlo iterations (and the number of field sources) to be performed. During each Monte Carlo simulation the field source is constructed as a sum of num_waves randomly generated plane waves. The seed for the plane wave phase shift and polarization is randomly chosen assigning to seed the value -1. A zero or positive seed value can be used to fix  the seed to the given number and to obtain consistently reproducible Monte Carlo iterations.

The generate_field_source controls if the field sources have to be generated and imported in the project. Otherwise only the statistical analysis of the field homogeneity will be performed and inserted as result into the 1D Result tree.

The option delete_previous_results removes all previously generated results and field sources, in order to reduce the required disk space and memory.

Finally the option user_probes evaluates for each electric (magnetic, respectively) field probe defined in the project and for each Monte Carlo iteration the field generated by the reverberation chamber.
